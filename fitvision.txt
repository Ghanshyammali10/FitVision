PART 1 — SIGNALING SERVER (server/server.js)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Build a Node.js + Express + Socket.io signaling server.

Dependencies: express, socket.io, cors, dotenv

The server must handle these Socket.io events:

CLIENT → SERVER:
- "join-room" ({ roomId, role })        → join a socket room, broadcast "user-joined" to others in the room
- "offer"   ({ roomId, offer })         → relay WebRTC offer to the other peer in the room
- "answer"  ({ roomId, answer })        → relay WebRTC answer to the other peer in the room
- "ice-candidate" ({ roomId, candidate }) → relay ICE candidate to the other peer
- "leave-room" ({ roomId })             → notify other peer, clean up

SERVER → CLIENT:
- "user-joined"    ({ role })           → tells existing peer that someone joined
- "offer"          ({ offer })          → forwarded offer from other peer
- "answer"         ({ answer })         → forwarded answer from other peer
- "ice-candidate"  ({ candidate })      → forwarded ICE candidate
- "user-left"      ()                   → other peer disconnected
- "room-full"      ()                   → room already has 2 people

Rooms object in memory: { roomId: [socketId1, socketId2] }
Max 2 people per room.
Generate room IDs as 6-character alphanumeric strings.
Add a "create-room" event that generates and returns a new room ID.

Server listens on PORT from .env, default 3001.
Add CORS for all origins in development.

PART 6 — WEBRTC CONNECTION (client/js/webrtc.js)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Use the native RTCPeerConnection API. No third-party WebRTC libraries.

ICE servers config:
{
  iceServers: [
    { urls: "stun:stun.l.google.com:19302" },
    { urls: "stun:stun1.l.google.com:19302" }
  ]
}

Connection flow:
1. Both seller and buyer join the same socket room
2. When buyer joins, server emits "user-joined" to seller
3. Seller creates RTCPeerConnection, adds local stream tracks
4. Seller creates offer → setLocalDescription → emits "offer" to server
5. Server relays offer to buyer
6. Buyer receives offer → setRemoteDescription → creates answer → setLocalDescription → emits "answer"
7. Server relays answer to seller
8. Seller receives answer → setRemoteDescription
9. Both sides exchange ICE candidates via "ice-candidate" events
10. ontrack event fires → set remote video element's srcObject

Export these functions from webrtc.js:
- initWebRTC(localStream, onRemoteStream)
- createOffer(roomId)
- handleOffer(offer, roomId)
- handleAnswer(answer)
- handleIceCandidate(candidate)
- closeConnection()

Handle disconnection: on "user-left" socket event, reset peer connection, clear remote video.

PART 7 — AI FEATURES (client/js/ai.js)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Load models from CDN (jsDelivr). All models run in-browser via TensorFlow.js.

Import via script tags:
- https://cdn.jsdelivr.net/npm/@tensorflow/tfjs@4.10.0/dist/tf.min.js
- https://cdn.jsdelivr.net/npm/@tensorflow-models/body-segmentation@1.0.1/dist/body-segmentation.js
- https://cdn.jsdelivr.net/npm/@tensorflow-models/pose-detection@2.1.3/dist/pose-detection.js
- https://cdn.jsdelivr.net/npm/@tensorflow-models/face-detection@1.0.2/dist/face-detection.js

FUNCTION 1 — removeBackground(videoElement) → returns ImageData
  - Use bodySegmentation.createSegmenter with MediaPipeSelfieSegmentation
  - runtime: "tfjs", modelType: "general"
  - Draw video frame to offscreen canvas
  - Run segmentPeople()
  - toBinaryMask() with threshold 0.5
  - Loop through pixels: if mask alpha < 128, set pixel alpha = 0
  - Return resulting canvas as PNG dataURL

FUNCTION 2 — getPose(videoElement) → returns keypoints array
  - Use poseDetection.createDetector with MoveNet SINGLEPOSE_LIGHTNING
  - Call estimatePoses(videoElement) every 100ms
  - Return poses[0].keypoints (array of {name, x, y, score})
  - Cache last valid pose; if no pose detected, return cached

FUNCTION 3 — getFace(videoElement) → returns { box: {xMin, yMin, width, height} }
  - Use faceDetection.createDetector with MediaPipeFaceDetector
  - runtime: "tfjs", maxFaces: 1
  - Call estimateFaces(videoElement)
  - Return faces[0] or null

FUNCTION 4 — drawAROverlay(ctx, canvasW, canvasH, garmentImg, poseKeypoints)
  - Extract keypoints: leftShoulder [5], rightShoulder [6], leftHip [11], rightHip [12]
  - Validate: both shoulders must have score > 0.25
  - Calculate placement (see Part 5 section C above)
  - ctx.globalAlpha = 0.85
  - ctx.drawImage(garmentImg, x, y, width, height)
  - ctx.globalAlpha = 1.0

FUNCTION 5 — drawMannequinFaceSwap(canvas, garmentImg, buyerVideoElement, faceData)
  - Draw body silhouette on canvas using SVG-style canvas drawing:
    - Head: circle at (cx, 70)
    - Neck: small rect
    - Torso: trapezoid (wider at shoulders, narrower at hips)
    - Arms: parallelogram shapes
    - Legs: two rects
    - Color: #2a2a3a (dark neutral)
  - If faceData exists:
    - Create offscreen canvas, draw buyerVideoElement frame
    - Crop box region from frame
    - ctx.save() → ctx.beginPath() → arc clip at head position → ctx.clip()
    - Draw cropped face into head circle
    - ctx.restore()
  - Overlay garmentImg on torso area (x = cx-70, y = headBottom+neckH, w = 140, h = 170)

IMPORTANT IMPLEMENTATION NOTES
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. All TF.js model loading must show a loading overlay with progress messages.
   Do not hide the loading screen until ALL three models are loaded.

2. Camera permissions: request camera+mic only after user clicks a button, 
   not on page load. Mobile browsers block autoplay of getUserMedia.

3. On seller mobile: default to facingMode "environment" (rear camera).
   On buyer desktop: default to facingMode "user" (front/selfie camera).

4. Garment images must be sent as compressed JPEG data URLs (quality 0.8) 
   to avoid large payloads over socket. Use canvas.toDataURL("image/jpeg", 0.8).

5. The AR overlay canvas must be positioned absolute on top of the video element,
   with pointer-events: none so the user can still interact with controls below.

6. The pose detection loop (requestAnimationFrame) must be stopped when:
   - The buyer switches away from the AR try-on tab
   - The camera is turned off
   - The page is hidden (use document.visibilitychange event)
   This prevents memory leaks and battery drain on mobile.

7. For the face swap, always check if faceData is fresh (< 2 seconds old).
   Use a timestamp on each faceData update. If stale, show placeholder face instead.

8. WebRTC renegotiation: if the seller switches camera (front/rear), 
   call sender.replaceTrack() instead of creating a new peer connection.

9. Error handling for all socket events — if "room-full" is received, 
   show a clear error: "This room is full. Ask the seller for a new link."

10. Add a "demo mode" fallback: if getUserMedia fails (permission denied or 
    no camera), load sample video files and simulate the full flow so the 
    app can still be demoed without a real camera.
